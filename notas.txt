Ahora que ya hice varios cambios en Git usaré éste documento para prácticar, y de paso hacer mis notas.

Primero que nada es necesario instalar y configurar adecuadamente Git

El siguiente paso a seguir es crear un nombre de usuario y mail para que se guarde como referencia de los commit

Posteriormente es necesario generar una clave SSH, que ya ví, pero tendré que repasar mas adelante.
(Se entiende como clave SSH como información cifrada que nos permite encriptar información y tener control de quién puede leer la información que enviamos mediante Git)

Ahora lo proximo que haremos es iniciar en la carpeta que sera nuestro entorno de trabajo principal con el comando "git init"
Este comando lo que hará es crear el "master" del proyecto en git, y a partir de aquí controlar los cambios. 

Hecho ésto debemos entender que hay 3 partes del proceso de trabajar en un proceso, y cada parte de ese proceso tiene un "espacio".

Esto es para mantener un contról de los cambios y evitar contratiempos, y al mismo tiempo supervisar adecuadamente los cambios.
Los espacios de trabajo son:
Directorio: Que es el espacio de trabajo del desarrollador, donde solo él puede ver los cambios.
Staging: Es el espacio en RAM que guarda los archivos modificados y los tiene listos antes de enviarse al proyecto principal.
Repositorio: Este podría considerarse como el proyecto principal y es la versión "definitiva" de los cambios añadidos del Staging.

Recordando que para iniciar el proyecto en Git, hay que usar el comando "git init" en el directorio raíz, ahora hay que revisar los comandos para trabajar en staging y el respositorio.

Una vez modificados los archivos en el directorio, para que los cambios pasen al staging hay que utilizar los siguientes comandos:
"git add (nombre y extensión del archivo)" ó "git add ." para añadir todos los archivos que se encuentren en la carpeta actual.

Parar verificar el estatus de nuestro staging usamos el comando "git status". Esto nos mostrará si hay archivos con modificaciones que podamos añadir al staging.
Si éste es el caso hay que ejecutar el comando "git add" para enviar esos cambios al staging.

Una vez que esten los cambios que consideremos pertinentes en el staging y querramos añadirlos al repositorio haremos un "commit"

La manera de realizar éste commit es con el comando "git commit -m "mensaje descriptivo del cambio"
Aqui es necesario aclarar, que aunque hay maneras de forzar un commit sin usar un mensaje descriptivo, es una MALA PRACTICA, además de que Git, nos dará advertencias para NO hacerlo.

Una vez hecho el commit, los cambios se borrarán del staging, y ahora formarán parte del repositorio y podran ser consultados por cualquiera con acceso a él.

Para verificar los cambios realizados a un repositorio, incluyendo el historial completo de los mismos usaremos el comando "git log"
Esto arrojará información de la fecha, hora, usuario, cambios y una clave ligada a dichos cambios, para asi poder analizar dichos cambios.

Al usar "git log" es necesario especificar que archivo es el que queremos examinar, de lo contrario se veran reflejados los cambios de todos los archivos cambiados simultaneamente.
Ej si quiero consultar cambios en el archivo estilos.css mi comando será "git log estilos.css"

Una vez elegido el archivo, el comando utilizado para comparar las distintas versiones será: "git diff"

La manera de usar éste comando es introducir git diff (serie númerica del cambio en git) (serie númerica del cambio en git)
El orden en que se introduscan las series numericas debe ser del mas antiguo al mas novedoso, ya que así es como GIT interpretará los cambios.
Es decir, si comparamos el mas viejo con el mas nuevo, nos dirá que añadimos al codigo, pero si primero introducimos el archivo nuevo, y luego el viejo, nos dirá que borramos cosas dentro del codigo, aunque ésto no haya sido así.

Existen 2 comandos para "retroceder" dentro versiones de un archivo y son "reset" y "checkout"
La diferencia entre ellos es que con checkout podremos consultar momentaneamente un archivo, pero en cualquier momento podremos devolver el respositorio a su estado, ANTES de que nosotros lo modifiquemos.
Es decir, si estoy en la versión 3 y quiero consultar la versión 2, hago "git checkout (serie numerica de versión) (nombre del archivo a consultar)"
Esto cambiara el repositorio y convertira la versión 3 en la versión 2, y cuando termine de revisar, solo necesito hacer el mismo comando, pero ésta vez seleccionando la serie numerica correspondiente a la versión 3

El segundo comando "reset" es sumamente peligroso, ya que, a diferencia de chekcout, no nos permite deshacer cambios. 
Si seleccionamos una version antigua del programa, nos situará en dicha versión, pero borrará PERMANENTEMENTE todas las versiones posteriores a la versión seleccionada.
La forma de implementarla sería: "git reset (serie numerica de versión) (nombre de archivo)"
Nuevamente, usar "reset" borrará PERMANENTEMENTE todos los cambios posteriores a la versión seleccionada.

Existe una opción adicional para git que es "rm", pero ésta opción afecta al staging y a l disco duro en el que estamos trabajando.

Hay 2 maneras de implementarlo:
git rm --cached: Elimina los archivos del área de Staging y del próximo commit pero los mantiene en nuestro disco duro.
git rm --force: Elimina los archivos de Git y del disco duro. Git siempre guarda todo, por lo que podemos acceder al registro de la existencia de los archivos, de modo que podremos recuperarlos si es necesario (pero debemos usar comandos más avanzados).

Retomando el git reset existen 3 formas de usarse que son:
git reset --soft: Borramos todo el historial y los registros de Git pero guardamos los cambios que tengamos en Staging, así podemos aplicar las últimas actualizaciones a un nuevo commit.
git reset --hard: Borra todo. Todo todito, absolutamente todo. Toda la información de los commits y del área de staging se borra del historial.

Y finalmente git reset HEAD: Este es el comando para sacar archivos del área de Staging. No para borrarlos ni nada de eso, solo para que los últimos cambios de estos archivos no se envíen al último commit, a menos que cambiemos de opinión y los incluyamos de nuevo en staging con git add, por supuesto.
